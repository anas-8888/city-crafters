## 1. ما هو هذا المشروع؟

نظام محفظة رقمية (مثل فودافون كاش أو Apple Pay مصغّر) يتيح:
- إنشاء محفظة لمستخدم
- معرفة رصيده
- تحويل أموال من مستخدم لآخر بشكل **آمن وغير متزامن**

### لماذا "غير متزامن"؟

لأنه لو جاء 1000 طلب تحويل في نفس الثانية، ومعالجتهم فورية، سيحدث تنافس على قاعدة البيانات وقد يُخصم الرصيد أكثر من مرة (double spending). الحل: نضع الطلبات في **طابور (Queue)** ونعالجها واحداً واحداً بشكل آمن.

## 2. التقنيات المستخدمة وكيف تعمل داخلياً

### 2.1 NestJS

**ما هو؟**
إطار عمل (Framework) لبناء تطبيقات Node.js باستخدام TypeScript. مبني على Express تحت الغطاء لكن يضيف هيكلاً منظماً.

**كيف يعمل داخلياً؟**
NestJS يعتمد على مفهوم **Dependency Injection (DI)**:
- أنت تُعرّف الـ Services والـ Controllers
- NestJS يُنشئ instance منها تلقائياً ويُوصّل كل service بمن يحتاجها
- مثال: `WalletController` يحتاج `WalletService` — NestJS يُعطيه إياها تلقائياً بدون أن تكتب `new WalletService()`

**المكونات الأساسية:**
- **Module**: مجموعة من الـ Controllers والـ Services المرتبطة ببعض — مثل مجلد واحد للمحفظة
- **Controller**: يستقبل الـ HTTP requests ويُعيد الـ responses
- **Service**: يحتوي على منطق العمل (business logic) — لا يتعامل مع HTTP مباشرة
- **DTO (Data Transfer Object)**: كلاس يُعرّف شكل البيانات القادمة في الـ request body ويتحقق منها

**الـ Decorators:**
الرموز التي تبدأ بـ `@` مثل `@Controller()`, `@Get()`, `@Post()` هي **decorators** — وظيفتها إخبار NestJS بدور كل كلاس أو دالة.

### 2.2 TypeORM

**ما هو؟**
مكتبة تُتيح التعامل مع قاعدة البيانات بكتابة TypeScript بدلاً من SQL مباشرةً. تُسمى ORM: **Object Relational Mapper** — تُحوّل الكلاسات في الكود إلى جداول في قاعدة البيانات.

**كيف يعمل داخلياً؟**
- تُعرّف **Entity** (كلاس يمثّل جدولاً)
- TypeORM يقرأ الـ decorators مثل `@Entity()`, `@Column()` ويُنشئ الجدول تلقائياً
- تستخدم `Repository` للقراءة والكتابة — كل Entity لها Repository خاص بها
- `synchronize: true` يعني TypeORM يُقارن الكود بقاعدة البيانات ويضيف/يعدّل الجداول تلقائياً (للتطوير فقط)

**مثال:**
```typescript
// هذا الكلاس يُنشئ جدول اسمه "wallets" في MySQL
@Entity('wallets')
class Wallet {
  @Column() userId: string;    // عمود اسمه user_id
  @Column() balance: string;   // عمود اسمه balance
}
```

### 2.3 MySQL

**ما هو؟**
قاعدة بيانات علائقية (Relational Database) تخزّن البيانات في جداول مترابطة.

**لماذا MySQL وليس MongoDB؟**
لأن البيانات المالية تحتاج:
- **ACID transactions**: إما أن تتم العملية كاملة أو لا تتم أصلاً
- **Row-level locking**: قفل سطر واحد في قاعدة البيانات بدلاً من الجدول كله
- **DECIMAL type**: تخزين الأرقام العشرية بدقة مطلقة (Float يُسبب أخطاء تقريب)

**InnoDB Engine:**
MySQL يستخدم InnoDB كـ storage engine افتراضي، وهو الذي يدعم:
- الـ Transactions
- الـ Foreign Keys
- الـ Row-level locking (SELECT FOR UPDATE)

---

### 2.4 Redis

**ما هو؟**
قاعدة بيانات In-Memory (تخزّن في الذاكرة RAM وليس على القرص) تعمل بسرعة فائقة. تُستخدم للـ caching والـ queues والـ pub/sub.

**كيف يُستخدم في هذا المشروع؟**
يُستخدم فقط كـ **backend للـ BullMQ** — أي BullMQ يُخزّن الـ Jobs في Redis. لما تُضيف job جديدة، تُكتب في Redis. لما يُعالجها الـ Worker، تُحذف أو تُحدَّث في Redis.

---

### 2.5 BullMQ

**ما هو؟**
مكتبة لإدارة **Job Queues** (طوابير المهام) مبنية على Redis.

**كيف تعمل داخلياً؟**

```
[API] ---> يضيف Job ---> [Redis Queue] ---> [Worker] يسحب Job ويعالجها
```

**دورة حياة الـ Job:**
1. `waiting`: الـ Job في الطابور تنتظر
2. `active`: الـ Worker يعالجها الآن
3. `completed`: تمّت بنجاح
4. `failed`: فشلت (يُعاد المحاولة تلقائياً حسب الإعداد)

**لماذا BullMQ وليس BullMQ مباشرة؟**
نستخدم `@nestjs/bull` وهو wrapper يُدمج BullMQ مع NestJS ويجعل الاستخدام أسهل.

**الـ Retry والـ Backoff:**
- `attempts: 3` → إذا فشلت الـ Job، يُعيد المحاولة 3 مرات
- `backoff: { type: 'exponential', delay: 2000 }` → ينتظر 2ث ثم 4ث ثم 8ث بين المحاولات (لا يُعيد المحاولة بشكل متواصل)

---

### 2.6 Docker و Docker Compose

**ما هو Docker؟**
Docker يُنشئ **containers** — بيئات معزولة تشغّل التطبيق مع كل اعتمادياته بغض النظر عن نظام التشغيل المضيف.

**ما هو الـ Dockerfile؟**
ملف يحتوي على تعليمات لبناء image (صورة) للتطبيق:
- من أين تبدأ (base image)
- ما الذي تنسخ
- كيف تبني الكود
- كيف تُشغّله

**ما هو Docker Compose؟**
أداة تُشغّل عدة containers معاً وتُعرّف العلاقات بينها (من يتصل بمن، من يعتمد على من).

**الـ Multi-stage Build:**
في الـ Dockerfile نستخدم مرحلتين:
- **builder**: تثبّت كل الـ packages (بما فيها devDependencies) وتبني الكود
- **production**: تأخذ فقط الـ output المبني وتثبّت production packages فقط → حجم أصغر بكثير

---

### 2.7 Winston (Logging)

**ما هو؟**
مكتبة logging تُخرج logs بتنسيق JSON منظّم مع timestamp.

**لماذا JSON؟**
في بيئة الـ containers، الـ logs تذهب إلى stdout ثم تُجمع بأدوات مثل Datadog أو CloudWatch. هذه الأدوات تفهم JSON وتُتيح البحث والفلترة.

**مثال الـ log:**
```json
{"level":"log","message":"Wallet created","userId":"user-alice","walletId":"abc-123","timestamp":"2024-01-15T10:30:00.000Z"}
```

### 📄 `tsconfig.json` — إعدادات TypeScript

```json
{
  "compilerOptions": {
    "module": "commonjs",
```
**`module: "commonjs"`**: يُحدد كيف يتم تقسيم الكود. CommonJS هو نظام Node.js التقليدي (`require/module.exports`). TypeScript سيُحوّل `import/export` إلى `require/module.exports` تلقائياً.

```json
    "emitDecoratorMetadata": true,
    "experimentalDecorators": true,
```
هذان مطلوبان لـ NestJS. الـ **Decorators** (`@Module()`, `@Injectable()` إلخ) ميزة TypeScript تجريبية. `emitDecoratorMetadata` يُخبر TypeScript يُضيف معلومات النوع في الكود المُجمَّع (مطلوب للـ Dependency Injection).

```json
    "target": "ES2021",
```
أيّ نسخة JavaScript ستُنتج؟ ES2021 تدعم `async/await` وكل الميزات الحديثة.

```json
    "baseUrl": "./",
    "ignoreDeprecations": "5.0",
```
**`baseUrl`**: نقطة البداية لحل المسارات غير النسبية. مطلوب لكي يعمل `@app/common`.
**`ignoreDeprecations`**: `baseUrl` أُعلن عن deprecation في TypeScript 6+، هذا السطر يُسكت التحذير رسمياً.

```json
    "strict": true,
    "strictNullChecks": false,
    "noImplicitAny": false,
```
- `strict: true`: يُفعّل مجموعة من الفحوصات الصارمة
- `strictNullChecks: false`: أوقفنا هذا لأن NestJS decorators أحياناً تُعيد `null` بشكل مقصود
- `noImplicitAny: false`: نسمح بـ `any` ضمنياً (NestJS يحتاجه)

```json
    "paths": {
      "@app/common": ["libs/common/src"],
      "@app/common/*": ["libs/common/src/*"]
    }
```
هذا **Path Alias** — بدلاً من كتابة:
```typescript
import { Wallet } from '../../../libs/common/src/entities/wallet.entity';
```
نكتب فقط:
```typescript
import { Wallet } from '@app/common';
```

### 📄 `docker-compose.yml` — تشغيل كل الـ Services معاً

```yaml
services:
  mysql:
    image: mysql:8.0
```
يُشغّل MySQL من الـ Docker Hub image الرسمية.

```yaml
    environment:
      MYSQL_ROOT_PASSWORD: ${DB_PASSWORD:-root}
      MYSQL_DATABASE: ${DB_NAME:-wallet_db}
```
`${DB_PASSWORD:-root}` تعني: خذ القيمة من متغير البيئة `DB_PASSWORD`، وإن لم يكن موجوداً استخدم `root` كقيمة افتراضية.

```yaml
    healthcheck:
      test: ['CMD', 'mysqladmin', 'ping', ...]
      interval: 10s
      retries: 10
```
Docker يُختبر هل MySQL جاهز كل 10 ثوانٍ. الـ API والـ Worker لن يبدآ إلا بعد أن يجتاز MySQL هذا الاختبار (`depends_on: condition: service_healthy`). هذا مهم لأن MySQL يحتاج وقتاً للتهيئة.

```yaml
  api:
    environment:
      DB_SYNCHRONIZE: 'true'
```
يُخبر TypeORM يُنشئ الجداول تلقائياً. في production حقيقية نضعها `false` ونستخدم migrations.

```yaml
    depends_on:
      mysql:
        condition: service_healthy
      redis:
        condition: service_healthy
```
الـ API لن يبدأ إلا بعد أن يكون MySQL وRedis جاهزَين.

### 📄 `libs/common/src/entities/wallet.entity.ts`

```typescript
@Entity('wallets')
export class Wallet {
```
`@Entity('wallets')` يُخبر TypeORM: هذا الكلاس يمثّل جدولاً اسمه `wallets`.

```typescript
  @PrimaryGeneratedColumn('uuid')
  id: string;
```
عمود `id` هو **Primary Key** يُولَّد تلقائياً بصيغة UUID (مثل `a1b2c3d4-e5f6-...`). UUID أفضل من الأرقام التسلسلية لأنه لا يكشف عدد السجلات ولا يسبب تعارضاً في البيئات الموزعة.

```typescript
  @Index({ unique: true })
  @Column({ name: 'user_id' })
  userId: string;
```
- `@Index({ unique: true })`: يُنشئ **Unique Index** في MySQL — يضمن أن كل userId فريد (مستخدم واحد = محفظة واحدة)، والـ Index يُسرّع البحث بـ userId
- `name: 'user_id'`: اسم العمود في قاعدة البيانات سيكون `user_id` (snake_case) بينما في الكود نستخدم `userId` (camelCase)

```typescript
  @Column({ type: 'decimal', precision: 18, scale: 2, default: '0.00' })
  balance: string;
```
**لماذا `decimal` وليس `float`؟**
- `float` يُخزّن الأرقام بصيغة ثنائية (binary) وينتج أخطاء تقريب: `0.1 + 0.2 = 0.30000000004`
- `decimal(18, 2)` يُخزّن كـ string عشري دقيق: يقبل حتى 18 رقماً، 2 منها بعد الفاصلة

**لماذا TypeScript نوعه `string` وليس `number`؟**
لأن TypeORM يُعيد DECIMAL columns كـ string من MySQL. نُحوّلها لـ `parseFloat()` عند الحاجة.

```typescript
  @CreateDateColumn({ name: 'created_at' })
  createdAt: Date;

  @UpdateDateColumn({ name: 'updated_at' })
  updatedAt: Date;
```
TypeORM يملأهما تلقائياً — `createdAt` عند الإنشاء، `updatedAt` عند كل تعديل.

### 📄 `libs/common/src/entities/transaction.entity.ts`

```typescript
export enum TransactionStatus {
  PENDING = 'pending',
  COMPLETED = 'completed',
  FAILED = 'failed',
}
```
**Enum** قائمة ثابتة من القيم. يمنع كتابة الـ status كـ string مفتوح وبالتالي يمنع الأخطاء الإملائية.

```typescript
  @Index()
  @Column({ name: 'idempotency_key', nullable: true, unique: true })
  idempotencyKey: string | null;
```
**Idempotency Key** مفتاح يُرسله العميل (مثل UUID فريد) مع كل طلب تحويل. إذا أعاد الإرسال نفس الـ Key (مثلاً بسبب انقطاع الشبكة)، النظام يُعيد نتيجة الطلب الأول بدلاً من تنفيذ تحويل ثانٍ.

- `nullable: true`: لأنه اختياري — العميل قد لا يُرسله
- `unique: true`: يمنع تكرار نفس الـ Key في قاعدة البيانات
- `@Index()`: يُنشئ index على هذا العمود لتسريع البحث

```typescript
  @Column({ name: 'failure_reason', type: 'text', nullable: true })
  failureReason: string | null;
```
لو فشل التحويل، نحفظ السبب هنا (مثل "رصيد غير كافٍ").

### 📄 `apps/api/src/main.ts`

```typescript
const app = await NestFactory.create(AppModule, { bufferLogs: true });
```
يُنشئ تطبيق NestJS. `bufferLogs: true` يحفظ الـ logs في buffer حتى يجهز الـ logger المخصص (Winston).

```typescript
app.useLogger(app.get(WINSTON_MODULE_NEST_PROVIDER));
```
يستبدل الـ logger الافتراضي لـ NestJS بـ Winston. `app.get()` يأخذ instance من أي service مسجّلة في الـ DI container.

```typescript
app.useGlobalPipes(
  new ValidationPipe({
    whitelist: true,        // يحذف أي حقول غير معرّفة في الـ DTO
    forbidNonWhitelisted: true,  // يرفض الطلب إذا جاءت حقول غير معروفة
    transform: true,        // يُحوّل البيانات القادمة للأنواع الصحيحة (string → number)
  }),
```

### 📄 `apps/api/src/transfer/transfer.service.ts`

```typescript
if (fromUserId === toUserId) {
  throw new BadRequestException('Sender and receiver must be different users');
}
```
تحقق بسيط بالكود — لا نذهب لقاعدة البيانات حتى.

```typescript
if (idempotencyKey) {
  const existing = await this.transactionRepo.findOne({ where: { idempotencyKey } });
  if (existing) {
    return { status: existing.status, transactionId: existing.id };
  }

```
**منطق الـ Idempotency:**
1. هل أرسل العميل `idempotencyKey`؟
2. هل يوجد transaction بهذا الـ key في قاعدة البيانات؟
3. إذا نعم → أعد النتيجة الموجودة بدلاً من إنشاء تحويل جديد

هذا يحمي من السيناريو: العميل أرسل الطلب، انقطع الاتصال، أعاد الإرسال — لا نُحوّل المال مرتين.

```typescript
const senderWallet = await this.walletRepo.findOne({ where: { userId: fromUserId } });
if (!senderWallet) throw new NotFoundException(...);

if (parseFloat(senderWallet.balance) < amount) {
  throw new BadRequestException('Insufficient balance');
}
```
**التحقق من الرصيد هنا = Optimistic Check (تحقق تفاؤلي)**

هذا التحقق **غير نهائي** — هو فقط لرفض الطلبات الواضحة الفشل بسرعة قبل إضافتها للطابور. التحقق الحقيقي يحدث في الـ Worker داخل lock.

```typescript
await this.transferQueue.add(TRANSFER_JOB, payload, {
  jobId: saved.id,     // معرّف فريد للـ Job = معرّف الـ Transaction
  attempts: 3,
  backoff: { type: 'exponential', delay: 2000 },
  removeOnComplete: false,
  removeOnFail: false,
});
```
- `jobId: saved.id`: لو أُضيفت نفس الـ Job مرتين (بنفس الـ ID)، BullMQ يتجاهل الثانية — طبقة حماية إضافية
- `removeOnComplete: false`: نحتفظ بسجلات الـ Jobs بعد اكتمالها للـ audit
- `removeOnFail: false`: نحتفظ بسجلات الـ Jobs الفاشلة للتحقيق

### 📄 `apps/worker/src/processors/transfer.processor.ts`

هذا أهم ملف في المشروع.

```typescript
@Processor(TRANSFER_QUEUE)
export class TransferProcessor {
```
`@Processor(TRANSFER_QUEUE)` يُخبر NestJS: هذا الكلاس مسؤول عن معالجة Jobs من الـ Queue اسمه `transfer-queue`.

```typescript

  async handleTransfer(job: Job<TransferJobPayload>): Promise<void> {
```
`@Process(TRANSFER_JOB)` يُخبر NestJS: هذه الدالة تُنفَّذ عند وصول Job نوعها `process-transfer`.

```typescript
    const existingTx = await txRepo.findOne({ where: { id: transactionId } });
    
    if (existingTx.status !== TransactionStatus.PENDING) {
      return; // idempotent – do nothing
    }
```
**Idempotency Guard في الـ Worker:**
لو كرّست المحاولة بعد crash جزئي، نتحقق: هل هذه الـ Transaction معالجة بالفعل؟ إذا نعم، نتجاهلها. هذا يمنع خصم الرصيد مرتين.

```typescript
    const queryRunner = this.dataSource.createQueryRunner();
    await queryRunner.connect();
    await queryRunner.startTransaction('READ COMMITTED');
```
**Database Transaction:**
`QueryRunner` يُتيح التحكم اليدوي في الـ transaction. `READ COMMITTED` مستوى العزل — يعني كل query ترى فقط البيانات المُعتمَدة (committed)، لا البيانات في الـ transactions الأخرى المفتوحة.

```typescript
    const [firstId, secondId] = [fromUserId, toUserId].sort();
```
**ترتيب الـ Locks لمنع الـ Deadlock:**

سيناريو الـ Deadlock:
- Job 1: تحويل من Alice لـ Bob → تقفل Alice أولاً، تنتظر Bob
- Job 2: تحويل من Bob لـ Alice → تقفل Bob أولاً، تنتظر Alice
- النتيجة: كلٌّ ينتظر الآخر → توقف تام (Deadlock)

الحل: نُرتّب الـ IDs أبجدياً دائماً قبل القفل:
- Job 1: `['Alice', 'Bob'].sort()` → تقفل Alice ثم Bob
- Job 2: `['Bob', 'Alice'].sort()` → تقفل Alice ثم Bob أيضاً
- النتيجة: كلاهما ينتظر Alice أولاً → الأول يأخذها والثاني ينتظر بأمان.

```typescript
    const firstWallet = await walletRepo
      .createQueryBuilder('wallet')
      .setLock('pessimistic_write')
      .where('wallet.userId = :userId', { userId: firstId })
      .getOne();
```
**`setLock('pessimistic_write')`** يُنفّذ:
```sql
SELECT * FROM wallets WHERE user_id = 'Alice' FOR UPDATE;
```

`FOR UPDATE` يقفل هذا السطر في قاعدة البيانات — أي transaction أخرى تحاول تعديل نفس السطر ستنتظر حتى ننتهي. هذا يمنع **Race Condition** تماماً.

```typescript
    const senderBalance = parseFloat(senderWallet.balance);
    if (senderBalance < amount) {
      throw new InsufficientFundsError(...);
    }
```
**التحقق الحقيقي من الرصيد** — هذا يحدث **داخل الـ lock**. لا يمكن لأي transaction أخرى تعديل الرصيد بين هذا التحقق والخصم.

```typescript
    await queryRunner.commitTransaction();
```
**Commit**: يُطبّق كل التغييرات دفعة واحدة — إما كلها أو لا شيء (Atomicity).

```typescript
    } catch (err) {
      await queryRunner.rollbackTransaction();
      
      const errorMessage = err instanceof Error ? err.message : String(err);
      
      if (isPermanent) {
        await txRepo.update(transactionId, {
          status: TransactionStatus.FAILED,
          failureReason: errorMessage,
        });
        return; // لا تُعيد الـ throw → BullMQ لن يُعيد المحاولة
      }
      
      throw err; // خطأ مؤقت → BullMQ سيُعيد المحاولة
    }
```
**تمييز نوع الخطأ:**
- **خطأ دائم** (InsufficientFundsError): نُحدّث الـ Transaction كـ `failed` ونُعيد `return` بدون throw → BullMQ لا يُعيد المحاولة (لا فائدة منها)
- **خطأ مؤقت** (DB crash، network error): نُعيد `throw` → BullMQ يُعيد المحاولة تلقائياً (قد يُصلح نفسه)

```typescript
    } finally {
      await queryRunner.release();
    }
```
`finally` يُنفَّذ دائماً سواء نجح أو فشل — يُعيد الـ connection للـ pool. لو نسينا هذا، ستنتهي connections قاعدة البيانات مع الوقت.

## 5. كيف تعمل الـ Flow من البداية للنهاية

### Flow الكامل لطلب تحويل:

```
المستخدم
  │
  │  POST /transfer { fromUserId, toUserId, amount }
  ▼
[API Controller] (transfer.controller.ts)
  │
  │  يُمرّر للـ Service
  ▼
[Transfer Service] (transfer.service.ts)
  │
  ├─ 1. تحقق: fromUserId ≠ toUserId
  ├─ 2. تحقق: idempotencyKey موجود مسبقاً؟ → أعد النتيجة الموجودة
  ├─ 3. جلب محفظة المُرسِل من DB → تحقق الرصيد (optimistic)
  ├─ 4. إنشاء Transaction بـ status = PENDING في DB
  ├─ 5. إضافة Job للـ Queue في Redis
  │
  │  يُعيد فوراً:
  │  { status: "processing", transactionId: "uuid" }
  ▼
المستخدم يستلم الرد الفوري

═══════════════════════════════════════
في الخلفية (بشكل مستقل):
═══════════════════════════════════════

[BullMQ] يُرسل الـ Job للـ Worker

[Worker] (transfer.processor.ts)
  │
  ├─ 1. يجلب الـ Transaction من DB
  ├─ 2. هل status = PENDING؟ لا → توقف (idempotency)
  ├─ 3. يفتح DB Transaction
  ├─ 4. يقفل محفظة Alice (FOR UPDATE)
  ├─ 5. يقفل محفظة Bob (FOR UPDATE)
  ├─ 6. يتحقق من الرصيد (authoritative)
  │     ├─ ناقص → يُلقي InsufficientFundsError
  │     └─ كافٍ → يخصم من Alice، يُضيف لـ Bob
  ├─ 7. يُحدّث Transaction status = COMPLETED
  ├─ 8. COMMIT
  └─ 9. يرفع الـ Locks
```
