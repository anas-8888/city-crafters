## 1. ما هو هذا المشروع؟

نظام محفظة رقمية (مثل فودافون كاش أو Apple Pay مصغّر) يتيح:
- إنشاء محفظة لمستخدم
- معرفة رصيده
- تحويل أموال من مستخدم لآخر بشكل **آمن وغير متزامن**

### لماذا "غير متزامن"؟

لأنه لو جاء 1000 طلب تحويل في نفس الثانية، ومعالجتهم فورية، سيحدث تنافس على قاعدة البيانات وقد يُخصم الرصيد أكثر من مرة (double spending). الحل: نضع الطلبات في **طابور (Queue)** ونعالجها واحداً واحداً بشكل آمن.

## 2. التقنيات المستخدمة وكيف تعمل داخلياً

### 2.1 NestJS

**ما هو؟**
إطار عمل (Framework) لبناء تطبيقات Node.js باستخدام TypeScript. مبني على Express تحت الغطاء لكن يضيف هيكلاً منظماً.

**كيف يعمل داخلياً؟**
NestJS يعتمد على مفهوم **Dependency Injection (DI)**:
- أنت تُعرّف الـ Services والـ Controllers
- NestJS يُنشئ instance منها تلقائياً ويُوصّل كل service بمن يحتاجها
- مثال: `WalletController` يحتاج `WalletService` — NestJS يُعطيه إياها تلقائياً بدون أن تكتب `new WalletService()`

**المكونات الأساسية:**
- **Module**: مجموعة من الـ Controllers والـ Services المرتبطة ببعض — مثل مجلد واحد للمحفظة
- **Controller**: يستقبل الـ HTTP requests ويُعيد الـ responses
- **Service**: يحتوي على منطق العمل (business logic) — لا يتعامل مع HTTP مباشرة
- **DTO (Data Transfer Object)**: كلاس يُعرّف شكل البيانات القادمة في الـ request body ويتحقق منها

**الـ Decorators:**
الرموز التي تبدأ بـ `@` مثل `@Controller()`, `@Get()`, `@Post()` هي **decorators** — وظيفتها إخبار NestJS بدور كل كلاس أو دالة.

### 2.2 TypeORM

**ما هو؟**
مكتبة تُتيح التعامل مع قاعدة البيانات بكتابة TypeScript بدلاً من SQL مباشرةً. تُسمى ORM: **Object Relational Mapper** — تُحوّل الكلاسات في الكود إلى جداول في قاعدة البيانات.

**كيف يعمل داخلياً؟**
- تُعرّف **Entity** (كلاس يمثّل جدولاً)
- TypeORM يقرأ الـ decorators مثل `@Entity()`, `@Column()` ويُنشئ الجدول تلقائياً
- تستخدم `Repository` للقراءة والكتابة — كل Entity لها Repository خاص بها
- `synchronize: true` يعني TypeORM يُقارن الكود بقاعدة البيانات ويضيف/يعدّل الجداول تلقائياً (للتطوير فقط)

**مثال:**
```typescript
// هذا الكلاس يُنشئ جدول اسمه "wallets" في MySQL
@Entity('wallets')
class Wallet {
  @Column() userId: string;    // عمود اسمه user_id
  @Column() balance: string;   // عمود اسمه balance
}
```

### 2.3 MySQL

**ما هو؟**
قاعدة بيانات علائقية (Relational Database) تخزّن البيانات في جداول مترابطة.

**لماذا MySQL وليس MongoDB؟**
لأن البيانات المالية تحتاج:
- **ACID transactions**: إما أن تتم العملية كاملة أو لا تتم أصلاً
- **Row-level locking**: قفل سطر واحد في قاعدة البيانات بدلاً من الجدول كله
- **DECIMAL type**: تخزين الأرقام العشرية بدقة مطلقة (Float يُسبب أخطاء تقريب)

**InnoDB Engine:**
MySQL يستخدم InnoDB كـ storage engine افتراضي، وهو الذي يدعم:
- الـ Transactions
- الـ Foreign Keys
- الـ Row-level locking (SELECT FOR UPDATE)

---

### 2.4 Redis

**ما هو؟**
قاعدة بيانات In-Memory (تخزّن في الذاكرة RAM وليس على القرص) تعمل بسرعة فائقة. تُستخدم للـ caching والـ queues والـ pub/sub.

**كيف يُستخدم في هذا المشروع؟**
يُستخدم فقط كـ **backend للـ BullMQ** — أي BullMQ يُخزّن الـ Jobs في Redis. لما تُضيف job جديدة، تُكتب في Redis. لما يُعالجها الـ Worker، تُحذف أو تُحدَّث في Redis.

---

### 2.5 BullMQ

**ما هو؟**
مكتبة لإدارة **Job Queues** (طوابير المهام) مبنية على Redis.

**كيف تعمل داخلياً؟**

```
[API] ---> يضيف Job ---> [Redis Queue] ---> [Worker] يسحب Job ويعالجها
```

**دورة حياة الـ Job:**
1. `waiting`: الـ Job في الطابور تنتظر
2. `active`: الـ Worker يعالجها الآن
3. `completed`: تمّت بنجاح
4. `failed`: فشلت (يُعاد المحاولة تلقائياً حسب الإعداد)

**لماذا BullMQ وليس BullMQ مباشرة؟**
نستخدم `@nestjs/bull` وهو wrapper يُدمج BullMQ مع NestJS ويجعل الاستخدام أسهل.

**الـ Retry والـ Backoff:**
- `attempts: 3` → إذا فشلت الـ Job، يُعيد المحاولة 3 مرات
- `backoff: { type: 'exponential', delay: 2000 }` → ينتظر 2ث ثم 4ث ثم 8ث بين المحاولات (لا يُعيد المحاولة بشكل متواصل)

---

### 2.6 Docker و Docker Compose

**ما هو Docker؟**
Docker يُنشئ **containers** — بيئات معزولة تشغّل التطبيق مع كل اعتمادياته بغض النظر عن نظام التشغيل المضيف.

**ما هو الـ Dockerfile؟**
ملف يحتوي على تعليمات لبناء image (صورة) للتطبيق:
- من أين تبدأ (base image)
- ما الذي تنسخ
- كيف تبني الكود
- كيف تُشغّله

**ما هو Docker Compose؟**
أداة تُشغّل عدة containers معاً وتُعرّف العلاقات بينها (من يتصل بمن، من يعتمد على من).

**الـ Multi-stage Build:**
في الـ Dockerfile نستخدم مرحلتين:
- **builder**: تثبّت كل الـ packages (بما فيها devDependencies) وتبني الكود
- **production**: تأخذ فقط الـ output المبني وتثبّت production packages فقط → حجم أصغر بكثير

---

### 2.7 Winston (Logging)

**ما هو؟**
مكتبة logging تُخرج logs بتنسيق JSON منظّم مع timestamp.

**لماذا JSON؟**
في بيئة الـ containers، الـ logs تذهب إلى stdout ثم تُجمع بأدوات مثل Datadog أو CloudWatch. هذه الأدوات تفهم JSON وتُتيح البحث والفلترة.

**مثال الـ log:**
```json
{"level":"log","message":"Wallet created","userId":"user-alice","walletId":"abc-123","timestamp":"2024-01-15T10:30:00.000Z"}
```

### 📄 `tsconfig.json` — إعدادات TypeScript

```json
{
  "compilerOptions": {
    "module": "commonjs",
```
**`module: "commonjs"`**: يُحدد كيف يتم تقسيم الكود. CommonJS هو نظام Node.js التقليدي (`require/module.exports`). TypeScript سيُحوّل `import/export` إلى `require/module.exports` تلقائياً.

```json
    "emitDecoratorMetadata": true,
    "experimentalDecorators": true,
```
هذان مطلوبان لـ NestJS. الـ **Decorators** (`@Module()`, `@Injectable()` إلخ) ميزة TypeScript تجريبية. `emitDecoratorMetadata` يُخبر TypeScript يُضيف معلومات النوع في الكود المُجمَّع (مطلوب للـ Dependency Injection).

```json
    "target": "ES2021",
```
أيّ نسخة JavaScript ستُنتج؟ ES2021 تدعم `async/await` وكل الميزات الحديثة.

```json
    "baseUrl": "./",
    "ignoreDeprecations": "5.0",
```
**`baseUrl`**: نقطة البداية لحل المسارات غير النسبية. مطلوب لكي يعمل `@app/common`.
**`ignoreDeprecations`**: `baseUrl` أُعلن عن deprecation في TypeScript 6+، هذا السطر يُسكت التحذير رسمياً.

```json
    "strict": true,
    "strictNullChecks": false,
    "noImplicitAny": false,
```
- `strict: true`: يُفعّل مجموعة من الفحوصات الصارمة
- `strictNullChecks: false`: أوقفنا هذا لأن NestJS decorators أحياناً تُعيد `null` بشكل مقصود
- `noImplicitAny: false`: نسمح بـ `any` ضمنياً (NestJS يحتاجه)

```json
    "paths": {
      "@app/common": ["libs/common/src"],
      "@app/common/*": ["libs/common/src/*"]
    }
```
هذا **Path Alias** — بدلاً من كتابة:
```typescript
import { Wallet } from '../../../libs/common/src/entities/wallet.entity';
```
نكتب فقط:
```typescript
import { Wallet } from '@app/common';
```

### 📄 `docker-compose.yml` — تشغيل كل الـ Services معاً

```yaml

    image: mysql:8.0
```
يُشغّل MySQL من الـ Docker Hub image الرسمية.

```yaml

```
`${DB_PASSWORD:-root}` تعني: خذ القيمة من متغير البيئة `DB_PASSWORD`، وإن لم يكن موجوداً استخدم `root` كقيمة افتراضية.

```yaml
      DB_SYNCHRONIZE: 'true'
```

```yaml
    depends_on:
      mysql:
        condition: service_healthy
      redis:
        condition: service_healthy
```
الـ API لن يبدأ إلا بعد أن يكون MySQL وRedis جاهزَين.

### 📄 `libs/common/src/entities/wallet.entity.ts`

```typescript
@Entity('wallets')
export class Wallet {
```
`@Entity('wallets')` يُخبر TypeORM: هذا الكلاس يمثّل جدولاً اسمه `wallets`.

```typescript
  @PrimaryGeneratedColumn('uuid')
  id: string;
```
عمود `id` هو **Primary Key** يُولَّد تلقائياً بصيغة UUID (مثل `a1b2c3d4-e5f6-...`). UUID أفضل من الأرقام التسلسلية لأنه لا يكشف عدد السجلات ولا يسبب تعارضاً في البيئات الموزعة.

```typescript
  @Index({ unique: true })
  @Column({ name: 'user_id' })
  userId: string;
```
- `@Index({ unique: true })`: يُنشئ **Unique Index** في MySQL — يضمن أن كل userId فريد (مستخدم واحد = محفظة واحدة)، والـ Index يُسرّع البحث بـ userId
- `name: 'user_id'`: اسم العمود في قاعدة البيانات سيكون `user_id` (snake_case) بينما في الكود نستخدم `userId` (camelCase)

```typescript
  @Column({ type: 'decimal', precision: 18, scale: 2, default: '0.00' })
  balance: string;
```
**لماذا `decimal` وليس `float`؟**
- `float` يُخزّن الأرقام بصيغة ثنائية (binary) وينتج أخطاء تقريب: `0.1 + 0.2 = 0.30000000004`
- `decimal(18, 2)` يُخزّن كـ string عشري دقيق: يقبل حتى 18 رقماً، 2 منها بعد الفاصلة

**لماذا TypeScript نوعه `string` وليس `number`؟**
لأن TypeORM يُعيد DECIMAL columns كـ string من MySQL. نُحوّلها لـ `parseFloat()` عند الحاجة.

```typescript
  @CreateDateColumn({ name: 'created_at' })
  createdAt: Date;

  @UpdateDateColumn({ name: 'updated_at' })
  updatedAt: Date;
```
TypeORM يملأهما تلقائياً — `createdAt` عند الإنشاء، `updatedAt` عند كل تعديل.

### 📄 `libs/common/src/entities/transaction.entity.ts`

```typescript
export enum TransactionStatus {
  PENDING = 'pending',
  COMPLETED = 'completed',
  FAILED = 'failed',
}
```
**Enum** قائمة ثابتة من القيم. يمنع كتابة الـ status كـ string مفتوح وبالتالي يمنع الأخطاء الإملائية.

```typescript
  @Index()
  @Column({ name: 'idempotency_key', nullable: true, unique: true })
  idempotencyKey: string | null;
```
**Idempotency Key** مفتاح يُرسله العميل (مثل UUID فريد) مع كل طلب تحويل. إذا أعاد الإرسال نفس الـ Key (مثلاً بسبب انقطاع الشبكة)، النظام يُعيد نتيجة الطلب الأول بدلاً من تنفيذ تحويل ثانٍ.

- `nullable: true`: لأنه اختياري — العميل قد لا يُرسله
- `unique: true`: يمنع تكرار نفس الـ Key في قاعدة البيانات
- `@Index()`: يُنشئ index على هذا العمود لتسريع البحث

```typescript
  @Column({ name: 'failure_reason', type: 'text', nullable: true })
  failureReason: string | null;
```
لو فشل التحويل، نحفظ السبب هنا (مثل "رصيد غير كافٍ").

### 📄 `apps/api/src/main.ts`

```typescript
const app = await NestFactory.create(AppModule, { bufferLogs: true });
```
يُنشئ تطبيق NestJS. `bufferLogs: true` يحفظ الـ logs في buffer حتى يجهز الـ logger المخصص (Winston).

```typescript
app.useLogger(app.get(WINSTON_MODULE_NEST_PROVIDER));
```
يستبدل الـ logger الافتراضي لـ NestJS بـ Winston. `app.get()` يأخذ instance من أي service مسجّلة في الـ DI container.

```typescript
app.useGlobalPipes(
  new ValidationPipe({
    whitelist: true,        // يحذف أي حقول غير معرّفة في الـ DTO
    forbidNonWhitelisted: true,  // يرفض الطلب إذا جاءت حقول غير معروفة
```

```typescript

    await queryRunner.connect();
    await queryRunner.startTransaction('READ COMMITTED');
```
**Database Transaction:**
`QueryRunner` يُتيح التحكم اليدوي في الـ transaction. `READ COMMITTED` مستوى العزل — يعني كل query ترى فقط البيانات المُعتمَدة (committed)، لا البيانات في الـ transactions الأخرى المفتوحة.

```typescript
    const [firstId, secondId] = [fromUserId, toUserId].sort();
```
**ترتيب الـ Locks لمنع الـ Deadlock:**

سيناريو الـ Deadlock:
- Job 1: تحويل من Alice لـ Bob → تقفل Alice أولاً، تنتظر Bob
- Job 2: تحويل من Bob لـ Alice → تقفل Bob أولاً، تنتظر Alice
- النتيجة: كلٌّ ينتظر الآخر → توقف تام (Deadlock)

الحل: نُرتّب الـ IDs أبجدياً دائماً قبل القفل:
- Job 1: `['Alice', 'Bob'].sort()` → تقفل Alice ثم Bob
- Job 2: `['Bob', 'Alice'].sort()` → تقفل Alice ثم Bob أيضاً
- النتيجة: كلاهما ينتظر Alice أولاً → الأول يأخذها والثاني ينتظر بأمان.

```typescript
    const firstWallet = await walletRepo
      .createQueryBuilder('wallet')
      .setLock('pessimistic_write')
      .where('wallet.userId = :userId', { userId: firstId })
      .getOne();
```
**`setLock('pessimistic_write')`** يُنفّذ:
```sql
SELECT * FROM wallets WHERE user_id = 'Alice' FOR UPDATE;
```

`FOR UPDATE` يقفل هذا السطر في قاعدة البيانات — أي transaction أخرى تحاول تعديل نفس السطر ستنتظر حتى ننتهي. هذا يمنع **Race Condition** تماماً.

```typescript
    const senderBalance = parseFloat(senderWallet.balance);
    if (senderBalance < amount) {
      throw new InsufficientFundsError(...);
    }
```
**التحقق الحقيقي من الرصيد** — هذا يحدث **داخل الـ lock**. لا يمكن لأي transaction أخرى تعديل الرصيد بين هذا التحقق والخصم.

```typescript
    await queryRunner.commitTransaction();
```
**Commit**: يُطبّق كل التغييرات دفعة واحدة — إما كلها أو لا شيء (Atomicity).

```typescript
    } catch (err) {
      await queryRunner.rollbackTransaction();
      
      const errorMessage = err instanceof Error ? err.message : String(err);
      
      if (isPermanent) {
        await txRepo.update(transactionId, {
          status: TransactionStatus.FAILED,
          failureReason: errorMessage,
        });
        return; // لا تُعيد الـ throw → BullMQ لن يُعيد المحاولة
      }
      
      throw err; // خطأ مؤقت → BullMQ سيُعيد المحاولة
    }
```
**تمييز نوع الخطأ:**
- **خطأ دائم** (InsufficientFundsError): نُحدّث الـ Transaction كـ `failed` ونُعيد `return` بدون throw → BullMQ لا يُعيد المحاولة (لا فائدة منها)
- **خطأ مؤقت** (DB crash، network error): نُعيد `throw` → BullMQ يُعيد المحاولة تلقائياً (قد يُصلح نفسه)

```typescript
      await queryRunner.release();

```
`finally` يُنفَّذ دائماً سواء نجح أو فشل — يُعيد الـ connection للـ pool. لو نسينا هذا، ستنتهي connections قاعدة البيانات مع الوقت.

```
  ├─ 4. إنشاء Transaction بـ status = PENDING في DB
  ├─ 5. إضافة Job للـ Queue في Redis
  │

  │  { status: "processing", transactionId: "uuid" }
  ▼
المستخدم يستلم الرد الفوري

═══════════════════════════════════════
في الخلفية (بشكل مستقل):
═══════════════════════════════════════

[BullMQ] يُرسل الـ Job للـ Worker

[Worker] (transfer.processor.ts)
  │
  ├─ 1. يجلب الـ Transaction من DB
  ├─ 2. هل status = PENDING؟ لا → توقف (idempotency)
  ├─ 3. يفتح DB Transaction
  ├─ 4. يقفل محفظة Alice (FOR UPDATE)
  ├─ 5. يقفل محفظة Bob (FOR UPDATE)

  │     └─ كافٍ → يخصم من Alice، يُضيف لـ Bob
  ├─ 7. يُحدّث Transaction status = COMPLETED
  ├─ 8. COMMIT
  └─ 9. يرفع الـ Locks
```

## 7. أسئلة متوقعة وإجاباتها

---

**س: لماذا استخدمت Monorepo بدلاً من مشروعين منفصلين؟**

ج: لأن الـ Monorepo يُتيح مشاركة الكود (`libs/common`) بين الـ API والـ Worker بدون نشره كـ npm package. لو كانا مشروعين منفصلين، كنا سنُكرّر تعريف الـ Entities أو نُنشئ package مشترك وندير نسخه.

---

**س: لماذا لم تستخدم `@nestjs/microservices` TCP/Redis transport؟**

ج: لأن طبيعة التواصل هنا **async one-way**: الـ API يُرسل job ولا ينتظر رداً فورياً. `@nestjs/microservices` مصمم للـ request/response المتزامن. BullMQ هو الأداة المناسبة لهذا الـ pattern بالذات.

---

**س: ما الفرق بين التحقق من الرصيد في الـ API والتحقق في الـ Worker؟**

ج: 
- **API (Optimistic)**: سريع، لكن غير نهائي — بين لحظة التحقق وتنفيذ الـ Job قد ينقص الرصيد (تحويل آخر)
- **Worker (Authoritative)**: داخل `SELECT FOR UPDATE` lock — لا يمكن لأي تحويل آخر تعديل الرصيد بينهما. هذا هو التحقق الحقيقي الذي يحمي من double spending

---

**س: ما هو Race Condition وكيف منعناه؟**

ج: Race Condition يحدث حين يتعامل عمليتان مع نفس البيانات في نفس الوقت بدون تنسيق.

مثال بدون حماية:
- Job 1 يقرأ رصيد Alice = 100
- Job 2 يقرأ رصيد Alice = 100
- Job 1 يخصم 80 → يكتب 20
- Job 2 يخصم 80 → يكتب 20 (خطأ! يجب أن يفشل لأن الرصيد 20 وليس 100)

مع `SELECT FOR UPDATE`:
- Job 1 يقفل صف Alice ويقرأ 100
- Job 2 يحاول يقفل Alice → ينتظر
- Job 1 يخصم 80، يكتب 20، يُعتمد، يرفع القفل
- Job 2 يقفل Alice الآن ويقرأ 20 → يفشل بـ InsufficientFunds

---

**س: ما هو Deadlock وكيف منعناه؟**

ج: Deadlock يحدث حين عمليتان كلٌّ منها تنتظر الأخرى:
- Transaction A تملك Lock على Alice وتنتظر Bob
- Transaction B تملك Lock على Bob وتنتظر Alice

الحل: **Consistent Lock Ordering** — نُرتّب الـ IDs أبجدياً ونقفل دائماً بنفس الترتيب. بالتالي كلا الـ transactions ستحاول تقفيل نفس الـ ID أولاً ولن يحدث circular wait.

---

**س: لماذا DECIMAL وليس FLOAT للرصيد؟**

ج: FLOAT يُخزّن الأرقام ثنائياً:
```
0.1 + 0.2 = 0.30000000000000004
```
DECIMAL(18,2) يُخزّن الأرقام بدقة مطلقة كـ fixed-point. لا يُقبل في أنظمة مالية أي خطأ تقريب.

---

**س: لماذا `balance` نوعه `string` في TypeScript وليس `number`؟**

ج: لأن TypeORM يُعيد DECIMAL columns من MySQL كـ string. لو جعلناها `number`، TypeScript ستقبل النوع لكن القيمة الفعلية في runtime ستكون string وستحصل على أخطاء غير متوقعة. نُبقيها string ونُحوّلها صراحةً بـ `parseFloat()` عند الحاجة.

---

**س: ما هو UUID ولماذا نستخدمه بدلاً من الأرقام التسلسلية (1, 2, 3...)?**

ج: UUID = Universally Unique Identifier مثل `a1b2c3d4-e5f6-7890-abcd-ef1234567890`.

لماذا أفضل:
- لا يكشف عدد السجلات (العدو لا يعرف كم transaction عندك)
- يعمل في البيئات الموزعة بدون تنسيق (كل سيرفر يولّد IDs فريدة)
- لا يمكن تخمين معرفات transactions أخرى

---

**س: ما معنى Idempotency؟**

ج: العملية الـ Idempotent هي التي يمكن تكرارها عدة مرات وتُعطي نفس النتيجة.

في سياقنا: لو أرسل العميل طلب تحويل، انقطع الاتصال، أعاد الإرسال بنفس `idempotencyKey` — النظام يُعيد نتيجة الطلب الأول بدلاً من تحويل المال مرتين.

---

**س: ماذا يحدث لو crash الـ Worker في منتصف التحويل؟**

ج: تتم العودة (Rollback) التلقائية:
1. الـ Worker يتوقف فجأة
2. MySQL يُلغي تلقائياً أي transaction غير مُعتمَدة (uncommitted)
3. الأرصدة تعود لما كانت عليه
4. BullMQ يُلاحظ أن الـ Job لم تكتمل ويُعيدها للطابور
5. الـ Worker يعود ويعالجها من البداية
6. الـ Idempotency guard يتحقق: `status = PENDING` → يُعيد المعالجة بأمان

---

**س: لماذا `restart: unless-stopped` في docker-compose؟**

ج: يعني لو crash الـ container (خطأ غير متوقع)، Docker يُعيد تشغيله تلقائياً. الاستثناء الوحيد: لو أوقفته أنت يدوياً بـ `docker compose down` فلا يُعيد التشغيل.

---

**س: ما الفرق بين `.env` و `.env.example`؟**

ج: 
- `.env.example`: نموذج يُضاف لـ Git — يُظهر للمطورين الآخرين ما هي المتغيرات المطلوبة بدون قيمها الحقيقية
- `.env`: الملف الفعلي بالقيم الحقيقية — **لا يُضاف لـ Git أبداً** (في `.gitignore`) لأنه قد يحتوي passwords وsecrets

---

**س: لو أردت نشر هذا في production حقيقية، ماذا يجب تغيير؟**

ج:
1. `DB_SYNCHRONIZE=false` وإنشاء migrations بدلاً منها
2. كلمات مرور قوية في `.env`
3. إضافة HTTPS (TLS)
4. إضافة Rate Limiting على الـ endpoints
5. إضافة Authentication (JWT)
6. استخدام managed MySQL وRedis (مثل AWS RDS و ElastiCache) بدلاً من Docker
7. إضافة monitoring (Prometheus/Grafana)

### 🔴 الحالة 1: Race Condition (التحويل المتزامن على نفس المحفظة)

#### ما المشكلة؟

تخيّل أن Alice رصيدها 100 ريال، وجاء طلبان في نفس الوقت:
- **طلب A**: تحويل 80 ريال من Alice
- **طلب B**: تحويل 80 ريال من Alice

بدون حماية:
```
Worker A → يقرأ رصيد Alice = 100 ✓ (يكفي)
Worker B → يقرأ رصيد Alice = 100 ✓ (يكفي)
Worker A → يخصم 80 → يكتب 20
Worker B → يخصم 80 → يكتب -60  ❌ (رصيد سالب!)
```

#### الحل: `SELECT FOR UPDATE` (Pessimistic Locking)

**الملف**: `apps/worker/src/processors/transfer.processor.ts`

```typescript
const fromWallet = await walletRepo.findOne({
  where: { userId: firstId },
  lock: { mode: 'pessimistic_write' },  // ← SELECT ... FOR UPDATE
});
```

`pessimistic_write` = `SELECT ... FOR UPDATE` في MySQL — يضع قفلاً على الصف حتى تنتهي الـ DB Transaction:

```
Worker A → يقفل صف Alice → يقرأ 100
Worker B → يحاول يقفل Alice → ينتظر... ⏳
Worker A → يخصم 80 → يكتب 20 → COMMIT → يرفع القفل
Worker B → يستيقظ، يقفل Alice الآن → يقرأ 20
Worker B → يتحقق: 80 > 20 ❌ → InsufficientFundsError
```

**النتيجة**: الطلب الثاني يفشل بشكل صحيح، لا يمكن لأحد أن ينفق أكثر مما يملك.

### 🔴 الحالة 2: Deadlock (تعطل متبادل بين عمليتين)

#### ما المشكلة؟

- **طلب A**: تحويل من Alice إلى Bob
- **طلب B**: تحويل من Bob إلى Alice

```
Worker A → يقفل Alice... ⏳ ثم يحاول يقفل Bob
Worker B → يقفل Bob...   ⏳ ثم يحاول يقفل Alice

كل واحد ينتظر الآخر → Deadlock → يتجمدان إلى الأبد ❌
```

#### الحل: Consistent Lock Ordering (ترتيب ثابت للأقفال)

**الملف**: `apps/worker/src/processors/transfer.processor.ts`

```typescript
const [firstId, secondId] = [fromUserId, toUserId].sort();

const fromWallet = await walletRepo.findOne({
  where: { userId: firstId },
  lock: { mode: 'pessimistic_write' },
});

const toWallet = await walletRepo.findOne({
  where: { userId: secondId },
  lock: { mode: 'pessimistic_write' },
});
```

`.sort()` يُرتّب الـ IDs أبجدياً — فلو كانت:
- `alice-id` = "aaa111"
- `bob-id` = "bbb222"

فكلا الـ Workers سيُقفلان "aaa111" أولاً دائماً، بغض النظر عن اتجاه التحويل:

```
Worker A (Alice→Bob): يقفل "aaa111"(Alice) أولاً، ثم "bbb222"(Bob)
Worker B (Bob→Alice): يقفل "aaa111"(Alice) أولاً ← ينتظر Worker A ينهي

لا circular wait → لا Deadlock ✅
```

### 🔴 الحالة 3: Worker Crash في منتصف التحويل

#### ما المشكلة؟

الـ Worker يبدأ التحويل، يخصم من Alice، ثم **يتعطل قبل أن يُضيف لـ Bob أو يُعيد الـ Job للطابور**. المال اختفى!

#### الحل: DB Transaction + BullMQ Retry

**الملف**: `apps/worker/src/processors/transfer.processor.ts`

```typescript
const queryRunner = this.dataSource.createQueryRunner();
await queryRunner.connect();
await queryRunner.startTransaction();

try {
  // خصم من Alice
  // إضافة لـ Bob
  // تحديث status = COMPLETED
  
  await queryRunner.commitTransaction();  // كل شيء أو لا شيء
} catch (err) {
  await queryRunner.rollbackTransaction();  // التراجع الكامل
  throw err;  // BullMQ يُعيد المحاولة
} finally {
  await queryRunner.release();  // إعادة الـ connection للـ pool
}
```

**ما يحدث عند الـ Crash:**
1. MySQL يُلغي تلقائياً أي transaction غير مُعتمَدة (Uncommitted) → الأرصدة تعود لأصلها
2. BullMQ يرى أن الـ Job لم تُكمَل → يُعيدها للطابور بعد فترة (exponential backoff)
3. الـ Worker يُعيد معالجتها من الصفر بأمان

**`finally` ضروري**: لو نسيناه ولم يُطلَق الـ connection، ستنضب connections قاعدة البيانات مع الوقت حتى يتجمد النظام.

### 🔴 الحالة 4: Idempotency (إرسال نفس الطلب مرتين)

#### ما المشكلة؟

- العميل يُرسل طلب تحويل
- انقطع الاتصال قبل أن يستلم الرد
- العميل يُعيد الإرسال بنفس البيانات

**بدون حماية: المال يُحوَّل مرتين.**

#### الحل: ثلاث طبقات حماية

**الطبقة 1 — Unique Index في DB:**

**الملف**: `libs/common/src/entities/transaction.entity.ts`

```typescript
@Column({ name: 'idempotency_key', unique: true, nullable: true })
idempotencyKey: string | null;
```

`unique: true` يمنع إدراج نفس الـ idempotencyKey مرتين في قاعدة البيانات — حتى لو وصل طلبان في نفس المللي ثانية، سيفشل أحدهما على مستوى DB.

---

**الطبقة 2 — Guard في الـ API:**

**الملف**: `apps/api/src/transfer/transfer.service.ts`

```typescript
if (dto.idempotencyKey) {
  const existing = await this.transactionRepo.findOne({
    where: { idempotencyKey: dto.idempotencyKey },
  });
  if (existing) {
    return existing;  // يُعيد النتيجة الموجودة بدون إنشاء جديدة
  }
}
```

لو وصل الطلب الثاني بعد نجاح الأول: يُعيد نتيجة الأول فوراً بدون معالجة.

---

**الطبقة 3 — Status Guard في الـ Worker:**

**الملف**: `apps/worker/src/processors/transfer.processor.ts`

```typescript
const tx = await transactionRepo.findOneBy({ id: transactionId });

if (tx.status !== TransactionStatus.PENDING) {
  return;  // تمت المعالجة مسبقاً → تجاهل
}
```

لو وصل نفس الـ Job للـ Worker مرتين (مثلاً بسبب BullMQ retry بعد crash): التحقق من الـ status يمنع إعادة التنفيذ.

**المعادلة**: الثلاث طبقات معاً تضمن أن كل تحويل حقيقي يُنفَّذ مرة واحدة بالضبط.

### 🔴 الحالة 5: Orphaned Transactions (transactions معلّقة بلا معالج)

#### ما المشكلة؟

```
API → يُنشئ Transaction بـ status=PENDING في DB ✓
API → يحاول يُضيف Job لـ Redis... ❌ Redis انقطع
```

Transaction موجودة في DB بـ `PENDING` لكن لا يوجد Job في Queue يعالجها. ستبقى معلّقة إلى الأبد.

#### الحل: Cron Job للمراقبة والإعادة

**الملف**: `apps/api/src/transfer/transfer-requeue.service.ts`

```typescript
@Cron(CronExpression.EVERY_5_MINUTES)
async requeueStuckTransactions(): Promise<void> {
  const fiveMinutesAgo = new Date(Date.now() - 5 * 60 * 1000);

  const stuck = await this.transactionRepo.find({
    where: {
      status: TransactionStatus.PENDING,
      createdAt: LessThan(fiveMinutesAgo),  // أُنشئت منذ أكثر من 5 دقائق
    },
  });

  for (const tx of stuck) {
    await this.transferQueue.add(TRANSFER_JOB, payload, {
      jobId: tx.id,   // ← نفس الـ jobId يمنع التكرار لو الـ Job موجودة فعلاً
    });
  }
}
```

**كل 5 دقائق:**
1. يبحث عن transactions بـ `status=PENDING` أنشئت منذ أكثر من 5 دقائق
2. يُعيد إضافتها للطابور بـ `jobId = transactionId`

**لماذا `jobId = transactionId`؟** BullMQ يرفض إضافة Job إذا كانت Job بنفس الـ ID موجودة مسبقاً. هذا يمنع إضافة نسخ مكررة في حال كانت الـ Job موجودة لكن تتأخر في المعالجة.

### 🟡 الحالة 6: Queue Flooding (إغراق الطابور)

#### ما المشكلة؟

مستخدم يُرسل آلاف الطلبات في ثانية واحدة:
- يملأ Redis بـ Jobs لا نهاية لها
- يُبطئ معالجة الطلبات الشرعية
- قد يُسبب نفاد ذاكرة Redis

#### الحل: Rate Limiting (تحديد معدل الطلبات)

**الملف**: `apps/api/src/app.module.ts`

```typescript
// تعريف الحد: 20 طلب كل 60 ثانية
ThrottlerModule.forRoot([{ ttl: 60_000, limit: 20 }]),
```

```typescript
// تطبيقه على جميع الـ endpoints تلقائياً

```

**كيف يعمل:**
- كل IP لديه عداد يُصفَّر كل 60 ثانية
- الطلب الـ 21 وما بعده يستلم `429 Too Many Requests`
- الطلبات الشرعية تعمل بشكل طبيعي

**لماذا 20 طلب؟** رقم مُعقول لمستخدم حقيقي (نادراً ما يُحوّل 20 مرة في دقيقة واحدة) لكنه يمنع الـ automated flooding.

### 🟡 الحالة 7: JavaScript Number Precision (دقة الأرقام الكبيرة)

#### ما المشكلة؟

JavaScript يستخدم 64-bit floating point (IEEE 754). الأرقام الكبيرة جداً تفقد دقتها:

```javascript
9999999999999999 === 10000000000000000  // true في JS! ❌
```

لو أرسل مستخدم `amount: 99999999999999999` كـ JSON، قد يُحوَّل لرقم مختلف قبل أن يصل للكود.

#### الحل: `@Max` Validator

**الملف**: `apps/api/src/transfer/dto/create-transfer.dto.ts`

```typescript
@IsNumber({ maxDecimalPlaces: 2 })
@Min(0.01, { message: 'amount must be greater than zero' })
@Max(1_000_000, { message: 'amount must not exceed 1,000,000' })
amount: number;
```

- `@Max(1_000_000)`: يرفض أي مبلغ فوق مليون → نبقى في النطاق الآمن لـ JavaScript
- `@Min(0.01)`: يرفض الصفر والأرقام السالبة
- `@IsNumber({ maxDecimalPlaces: 2 })`: يرفض أكثر من منزلتين عشريتين (لا يُقبل `10.999`)

### 🟡 الحالة 9: Duplicate Index Bug في TypeORM

#### ما المشكلة؟

في TypeORM، لو استخدمت **كلا من** `@Index()` و `unique: true` على نفس الـ column:

```typescript
@Index({ unique: true })            // ينشئ: IDX_xxxx
@Column({ unique: true })           // ينشئ: UQ_xxxx
idempotencyKey: string;
```

TypeORM ينشئ **مفتاحَين فريدَين مختلفَين** على نفس العمود. عند إعادة تشغيل التطبيق:

```
Error: Duplicate key name 'IDX_11a02d187c87d3dc5b0b4949f2'
```

لأن `unique: true` في `@Column` أنشأ الـ index الفعلي، وعند التشغيل الثاني يحاول TypeORM إنشاء `IDX_` مجدداً ليجده موجوداً.

#### الحل: الاكتفاء بـ `unique: true` فقط

**الملف**: `libs/common/src/entities/transaction.entity.ts`

```typescript
// ❌ خطأ
@Index({ unique: true })
@Column({ unique: true })
idempotencyKey: string;

// ✅ صحيح
@Column({ unique: true })  // هذا وحده يكفي لإنشاء UNIQUE KEY
idempotencyKey: string;
```

`unique: true` في `@Column` ينشئ `UNIQUE KEY` في MySQL وهو بحد ذاته index. لا حاجة لـ `@Index()` إضافي.

### 🟡 الحالة 10: TypeScript `unknown` Type في catch blocks

#### ما المشكلة؟

مع `strict: true` في TypeScript، متغير الـ `err` في catch block يكون نوعه `unknown` وليس `Error`:

```typescript
catch (err) {
  console.log(err.message);  // ❌ TypeScript Error: err is of type 'unknown'
}
```

#### الحل: Type Guard

**الملف**: `apps/worker/src/processors/transfer.processor.ts`

```typescript
catch (err) {
  const errorMessage = err instanceof Error ? err.message : String(err);
  // الآن errorMessage نوعه string بأمان ✅
}
```

`instanceof Error` يتحقق في runtime هل الـ err هو Error object — إذا نعم يستخدم `.message`، وإلا يُحوّله لـ string بـ `String()`.
